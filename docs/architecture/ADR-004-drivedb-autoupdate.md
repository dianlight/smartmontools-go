# ADR-004: Automated drivedb.h Tracking

## Status

Accepted

## Context

The library embeds `drivedb.h` from the [smartmontools project](https://github.com/smartmontools/smartmontools)
to resolve USB bridge device types without requiring a live smartctl installation.
The file is currently updated manually: a developer must notice a new smartmontools
release, download the file, and open a PR.

This manual process means:
- New USB bridges added to upstream drivedb go unrecognised until a maintainer acts.
- Updates may lag by weeks or months.
- There is no audit trail linking a specific drivedb revision to a library release.

For the future **IoctlBackend** and **NativeBackend**, `drivedb.h` is even more
critical: those backends use the database to determine device protocols without
falling back to `smartctl`. Stale data directly impacts compatibility.

---

## Decision

Automate drivedb.h synchronisation using a dedicated GitHub Actions workflow that:

1. **Polls** the upstream smartmontools repository daily for changes to `lib/drivedb.h`.
2. **Compares** the upstream file against the embedded copy using SHA-256.
3. **Opens a pull request** automatically when a new revision is detected.
4. **Embeds drivedb metadata** (upstream commit SHA + date) as a Go constant, so
   callers can inspect which revision is in use.

### Workflow: `.github/workflows/drivedb-update.yml`

```yaml
name: Update drivedb.h

on:
  schedule:
    - cron: '0 6 * * *'   # daily at 06:00 UTC
  workflow_dispatch:        # allow manual trigger

jobs:
  update-drivedb:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write

    steps:
      - uses: actions/checkout@v4

      - name: Fetch upstream drivedb.h
        id: fetch
        run: |
          UPSTREAM_URL="https://raw.githubusercontent.com/smartmontools/smartmontools/master/lib/drivedb.h"
          curl -fsSL "$UPSTREAM_URL" -o /tmp/drivedb_upstream.h
          UPSTREAM_SHA=$(sha256sum /tmp/drivedb_upstream.h | awk '{print $1}')
          LOCAL_SHA=$(sha256sum backends/exec/drivedb.h | awk '{print $1}')
          echo "upstream_sha=$UPSTREAM_SHA" >> "$GITHUB_OUTPUT"
          echo "local_sha=$LOCAL_SHA" >> "$GITHUB_OUTPUT"
          echo "changed=$([ "$UPSTREAM_SHA" != "$LOCAL_SHA" ] && echo true || echo false)" >> "$GITHUB_OUTPUT"

      - name: Fetch upstream commit metadata
        if: steps.fetch.outputs.changed == 'true'
        id: meta
        run: |
          COMMIT=$(curl -fsSL \
            "https://api.github.com/repos/smartmontools/smartmontools/commits?path=lib/drivedb.h&per_page=1" \
            | jq -r '.[0].sha')
          DATE=$(curl -fsSL \
            "https://api.github.com/repos/smartmontools/smartmontools/commits/$COMMIT" \
            | jq -r '.commit.committer.date')
          echo "commit=$COMMIT" >> "$GITHUB_OUTPUT"
          echo "date=$DATE" >> "$GITHUB_OUTPUT"

      - name: Update drivedb.h and version constant
        if: steps.fetch.outputs.changed == 'true'
        run: |
          cp /tmp/drivedb_upstream.h backends/exec/drivedb.h
          # Update the embedded metadata constants in drivedb_version.go
          cat > backends/exec/drivedb_version.go <<EOF
          package exec

          // DrivedbUpstreamCommit is the upstream smartmontools commit SHA
          // from which the embedded drivedb.h was taken.
          const DrivedbUpstreamCommit = "${{ steps.meta.outputs.commit }}"

          // DrivedbUpstreamDate is the commit date of DrivedbUpstreamCommit.
          const DrivedbUpstreamDate = "${{ steps.meta.outputs.date }}"
          EOF

      - name: Create Pull Request
        if: steps.fetch.outputs.changed == 'true'
        uses: peter-evans/create-pull-request@v6
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          commit-message: |
            🗃️ chore(drivedb): update drivedb.h to smartmontools@${{ steps.meta.outputs.commit }}
          title: "🗃️ chore(drivedb): update drivedb.h (${{ steps.meta.outputs.date }})"
          body: |
            Automated update of the embedded `drivedb.h` from upstream smartmontools.

            **Upstream commit**: smartmontools/smartmontools@${{ steps.meta.outputs.commit }}
            **Date**: ${{ steps.meta.outputs.date }}

            Changes were detected by comparing SHA-256 checksums:
            - Previous: `${{ steps.fetch.outputs.local_sha }}`
            - New: `${{ steps.fetch.outputs.upstream_sha }}`

            Please review the diff for any removed or changed entries that might
            affect existing USB bridge lookups.
          branch: drivedb/auto-update
          delete-branch: true
          labels: drivedb,dependencies,automated
```

### Embedded Version Constant

The file `backends/exec/drivedb_version.go` is generated by the workflow and committed
alongside `drivedb.h`. It allows runtime inspection via the root-package re-exports:

```go
// Example: log which drivedb revision is in use
log.Info("drivedb loaded",
    "commit", smartmontools.DrivedbUpstreamCommit,
    "date",   smartmontools.DrivedbUpstreamDate,
)
```

### Renovate / Dependabot as Alternative

[Renovate](https://docs.renovatebot.com/) supports custom datasources that track
commit SHAs. The `renovate.json` configuration includes:

- **`customDatasources.smartmontools-drivedb`**: Polls the GitHub commits API for
  `lib/drivedb.h` changes and surfaces the latest commit SHA as a Renovate version.
- **`customManagers` (regex)**: Scans `backends/exec/drivedb_version.go` for
  `DrivedbUpstreamCommit = "<sha>"` to determine the currently embedded version.
- **`packageRules`**: Scopes PR titles, labels, and branch names for drivedb updates.

When Renovate detects a new upstream SHA, it opens a PR updating only
`DrivedbUpstreamCommit` in `drivedb_version.go`. The companion workflow
`.github/workflows/drivedb-fetch.yml` is triggered by that PR and:
1. Reads the new SHA from `drivedb_version.go`.
2. Downloads the matching `drivedb.h` from upstream.
3. Updates `DrivedbUpstreamDate` from the GitHub commits API.
4. Commits both files back to the PR branch.

This approach is complementary to the standalone `drivedb-update.yml` workflow:
both may be active simultaneously without conflict (Renovate deduplicates
open PRs on the same branch).

### Validating Updated drivedb.h

The existing `loadDrivedbAddendum()` unit tests run against the embedded file.
Any PR that updates `drivedb.h` will trigger CI, which catches:
- Parse regressions (a new entry format that breaks the parser).
- Removed USB bridge entries (tests that use specific IDs will fail if entries
  were removed upstream).

## Migration Notes for IoctlBackend / NativeBackend

When the IoctlBackend uses drivedb.h to select device protocols, the drivedb
parser (`drivedb.go`) must be extended to also expose:

- Full drive model entries (not only USB entries), to identify ATA-specific
  firmware quirks and timeout overrides.
- Firmware regex matching, currently skipped by the USB-only parser.

This extension is tracked in the IoctlBackend implementation work, not in this ADR.

---

## Consequences

### Positive
- drivedb.h stays current with zero manual effort.
- PRs are reviewable and include a full upstream diff.
- Runtime version constants enable diagnostics and support queries.
- CI validates correctness of every update.

### Negative
- An extra workflow adds CI minutes (minimal: ~30s per run).
- `drivedb_version.go` is a generated file; the repo must document this clearly
  to avoid developers editing it by hand.

### Neutral
- The `drivedb.h` update cycle is independent of library releases; a drivedb PR
  can be merged and tagged as a patch release without any code changes.
